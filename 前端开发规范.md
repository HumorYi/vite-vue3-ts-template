# 前端开发规范

## 1. 命名规范

| 类型           | 命名规则                | 示例                          |
| -------------- | ----------------------- | ----------------------------- |
| 项目/文件夹    | kebab-case（小写横杠）  | user-info、order-list、utils  |
| 文件（组件）   | PascalCase（大驼峰）    | UserCard.vue、OrderDetail.vue |
| 文件（非组件） | kebab-case （小写横杠） | utils/request.ts、api/user.ts |
| 变量/函数      | camelCase（小驼峰）     | userList、getUserInfo()       |
| 常量           | 全大写+下划线           | MAX_COUNT、BASE_URL           |
| CSS类名        | kebab-case              | .btn-primary、.form-item      |

## 2. HTML / Vue 模板规范

- 标签自闭合：`<img />`、`<input />`，标签名/属性名全小写，属性值用双引号包裹
- 属性书写顺序（优先级从高到低）：ref → v-if → v-for → v-model → 事件 → class
- 模板仅做渲染，禁止编写复杂逻辑，复杂判断/计算抽离至computed/hooks/methods
- 事件绑定：使用 `@click="handleClick"` 形式，禁止内联逻辑（如 `@click="count++"`）
- v-for 必须搭配 :key，且key使用唯一标识（如ID），禁止使用索引

## 3. CSS / SCSS 规范

- Vue组件样式添加 `scoped` 属性，避免全局样式污染
- 选择器嵌套层级不超过 3 层，避免过度嵌套
- 单位统一：统一使用 px，移动端会自动转成 rem / vw
- 颜色、字号、间距等通用样式抽离为SCSS变量，统一维护
- 禁止滥用 `!important`，仅极特殊场景（如覆盖第三方组件）使用并备注原因

## 4. JavaScript / TypeScript 规范

- 变量声明：优先使用 `const`/`let`，禁用 `var`
- 语法特性：优先使用箭头函数、可选链（`?.`）、空值合并（`??`），示例：`user?.name ?? ''`
- 异步处理：统一使用 `async/await` 语法，禁止回调嵌套，记得加上 `try/catch`
- 类型约束：所有变量、函数入参/返回值必须显式定义类型，禁止使用 `any` 兜底
- 工具函数：通用工具函数统一存放至 `utils/` 目录，按功能拆分文件

## 5. Vue3 规范

- 脚本语法：统一使用 `<script setup lang="ts">` 语法糖，禁用 Options API
- Props 规则：必须定义类型 + 默认值，禁止使用 `any` 类型
- 逻辑拆分：业务逻辑按功能拆分为自定义 composables（命名规则：useXxx），存放至 `composables/` 目录
- 代码组织：变量、函数、computed 按逻辑分组（导入→Props→响应式数据→计算属性→方法→生命周期）
- 副作用清理：
  - 定时器、事件监听等 优先使用 @vueuse/core 提供的 composables，自动在 `onUnmounted` 中清理，避免内存泄漏
  - 网络请求 优先使用 composables/useApiOption， 自动在 `onUnmounted` 中中断请求，避免内存泄漏

## 6. 项目结构

```plaintext
src/
├── api/              接口请求（按模块拆分）
├── assets/           静态资源（图片、字体、图标、样式等）
├── components/       公共组件（同模块的提升到目录下，如 RouterLink/Permission.vue，最终调用为 <RouterLinkPermission /> ）
├── composables/      自定义 composables
├── config/           自定义 配置
├── directives/       自定义 指令
├── dts/              TypeScript 声明文件
├── layouts/          布局组件（同 公共组件）
├── pages/            页面组件（对应路由）
├── router/           路由配置
├── store/            Pinia状态管理
├── types/            TypeScript 类型声明
├── uno/              UnoCss 预设、快捷方式等
├── utils/            工具函数
```

## 7. Git 规范

### 7.1 分支命名

- 主分支：main（生产环境）、dev（开发环境）
- 功能分支：feature/xxx（示例：feature/user-list）
- 修复分支：fix/xxx（示例：fix/login-failure）

### 7.2 Commit 格式

格式要求：`type: 描述`（描述需简洁，字数≤50字）

示例：

- feat: 新增用户列表分页功能
- fix: 修复登录验证码失效问题
- style: 格式化代码，调整缩进
- docs: 更新接口文档
- refactor: 重构用户请求逻辑
- perf: 优化列表渲染性能
- test: 新增登录功能测试用例
- chore: 升级依赖包版本

## 8. 工程化配置（必配）

- 代码格式化：配置 ESLint + StyleLint + Prettier，统一代码语法和格式规则
- 提交校验：配置 Husky + lint-staged，提交代码前自动执行 代码格式化
- 日志约束：禁止将 `console.log`/`debugger` 提交到代码仓库
- 构建优化：开启代码压缩、Tree Shaking、图片优化，减小打包体积

### 总结

1. Git分支按用途分类命名，Commit信息需遵循`type: 描述`格式，保证提交记录清晰可追溯；
2. 工程化配置需强制落地，通过ESLint、StyleLint、Prettier统一格式，Husky拦截不规范提交，构建环节做好性能优化。
